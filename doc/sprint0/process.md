# Our Process

## **Team Organization**
The setup part of Sprint 0 included the following:
-  Setting up a clear workflow before implementation.
-  Division of responsibilities according to interest and availability.
-  The establishment of a light meeting structure (facilitator, note-taker, timekeeper) to ensure productivity and consistency.

To make responsibilities explicit while still working collaboratively, we informally assigned **work responsibilities**:
- **Documentation contributor:** drafted and refined Sprint 0 documents, and coordinated team edits.
- **Backlocg/user story contributors:** created and refined user stories in Trello, and ensured each story had acceptance criteria and a clear definition of done.
- **Repo/process contributors:** ensured GitHub repo organisation was clean, and that finalised deliverables were committed in the correct location.
This helped everyone contribute meaningfully (documentation, feature brainstorming, workflow setup), while still keeping accountability through task ownership.

To coordinate work, we used:
- **Discord** as our primary communication channel (quick questions, updates, reminders, suggestions, etc.).
- **Trello** to track tasks using a simple status flow (To do / In progress / Completed), as well as documenting our product backlog.
- **GitHub** for version control and to store finalized deliverables and documentation.

## Decision-Making
Decisions (tech stack, feature scope, task distribution, user stories) were made either through short discussions in meetings or through written communication in Discord. Our goal was to reach consensus whenever possible. When opinions differed, we resolved it by identifying the main trade-offs (scope, effort, risk, cost) and choosing the option on that best supported a realistic Sprint 1 plan.

## Prioritizing User Stories
We prioritized user stories using three main criteria:
1. **MVP value:** features required for the marketplace to function in its simplest usable form.
2. **Dependencies:** work that unlocks other work (e.g., data model choices before UI integration).
3. **Risk reduction:** tasks with uncertainty that we should address early to avoid surprises later.

### Estimation / Points & Concensus
For estimating stories, we used **relative sizing** (comparing complexity rather than predicting exact hours). We estimated using a shared point scale (e.g., **1 / 2 / 3 / 5 / 8**). Each member proposed an estimate, then we discussed differences in assumptions (edge cases, unclear requirements, technical unknowns). On average, it took **1–2 rounds** of voting/discussion to reach agreement on a point estimate, with more rounds only when the story scope was unclear.

## Meetings and Communication
We met, and will continue to meet **three times per week at 5pm EST** (Tuesdays, Thursdays, and Saturdays), unless rescheduled at least 24 hours in advance (this is subject to change depending on circumstance). Meetings were held virtually (Discord call) or in-person when practical. =

Our meeting structure was consistent:
- Check-in (To do / In progress / Completed)
- Review sprint board/backlog
- Assign tasks and confirm deadlines
- Identify blockers/risks
- Confirm next steps

## Moving Forward - Sprint 1 Plan
Sprint 0 reinforced a few important lessons we want to carry into Sprint 1:
- **Write decisions down:** stating key decisions in Discord increases efficiency.
- **Track tasks consistently:** actively update Trello and GitHub to prevent surprises and imnprove planning.
- **Clarify scope early:** defining a clear definition of done makes planning smoother.

Overall, Sprint 0 helped us align on expectations, define how we collaborate, and set a foundation for a more structured and efficient Sprint 1.
